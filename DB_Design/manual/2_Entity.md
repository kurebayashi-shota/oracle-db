# エンティティの洗い出し（オンラインショップ例）

## 目的

要件（ユーザー、商品、注文）を基に、DB で扱うテーブル単位に分ける段階です。

- 1 つのテーブル＝ 1 つのテーマ を原則にする

---

## 使用できるツール

- **テキストエディタ**（VS Code, Sublime Text など）

  - 箇条書きでテーブル候補を整理

- **スプレッドシート**（Excel, Google Sheets）

  - 「テーブル名」「用途・役割」「主な属性候補」を列に整理

- **マインドマップツール**（XMind, FreeMind など）
  - テーブルと関係性を視覚化すると理解しやすい

---

## 作業手順（PC 上での整理方法）

### 1. 管理対象からテーブル候補を抽出

要件定義で整理した「ユーザー」「商品」「注文」を基にテーブル候補を作る

例:

- ユーザー → `users`
- 商品 → `products`
- 注文 → `orders`

### 2. 必要に応じて補助テーブルも検討

1 つの注文に複数の商品を含める場合は中間テーブルを作る

例:

- 注文商品 → `order_items`（注文 ID と商品 ID を紐付ける）

### 3. 用途・役割を整理

テキストやスプレッドシートで表形式にまとめると便利

| テーブル名  | 用途・役割                                   |
| ----------- | -------------------------------------------- |
| users       | ユーザー情報管理（名前、メールアドレスなど） |
| products    | 商品情報管理（商品名、価格、在庫など）       |
| orders      | 注文情報管理（誰がいつ注文したか）           |
| order_items | 注文と商品を紐付け（数量や価格も記録）       |

---

## 確認ポイント

- 1 つのテーブルに異なるテーマが混ざっていないか
- 複数テーブル間の関係（1 対多、多対多）が整理できるか
- 将来的な拡張（配送情報、決済情報など）を追加しやすいか

---

## 💡 ポイント

- この段階ではまだカラムの詳細やデータ型は決めなくて OK
- 「テーブルの単位」と「役割」を明確にすることが重要
- 中間テーブルを作るかどうかも、この段階で決めると後の設計がスムーズ

# 設計例

## 管理対象からテーブル候補を抽出

- 個人情報 -> 'persons'
- 家族情報 -> 'familys'

## 必要に応じて補助テーブルも検討

- 所属履歴 -> 'history'

## 用途・役割を整理

| テーブル名 | 用途・役割                                                 |
| ---------- | ---------------------------------------------------------- |
| persons    | 個人情報管理（姓名、生年月日、結婚、子供の数、死亡、旧姓） |
| familys    | 家族情報管理（姓、所属個人）                               |
| history    | 所属情報管理（席の移動や離婚、再婚を表現）                 |
