# 正規化の検討（オンラインショップ版）

## 目的

- データの重複を減らし、整合性のある DB 構造にする
- 更新・削除・挿入の際に不整合が起きないようにする
- **第 1 正規形（1NF）～第 3 正規形（3NF）** を意識する

---

## 作業手順

### 1. 第 1 正規形（1NF）

- カラムは **単一値** であること
- 配列やカンマ区切りで値を持たせない

例:

- `users` テーブルの `name` は単一の文字列
- `products` の `price` は単一値
- `order_items` に複数の商品 ID をまとめて書かない → OK

---

### 2. 第 2 正規形（2NF）

- **主キーに完全依存しているか** を確認
- 複合キーの場合、部分依存している属性は別テーブルに分ける

例:

- `order_items` の主キーを `id` または `(order_id, product_id)` に設定
- `quantity` と `price` はそれぞれの注文と商品に完全依存 → OK

---

### 3. 第 3 正規形（3NF）

- **非キー属性同士で依存していないか** を確認
- 非キー属性間で依存があれば別テーブルに分ける

例:

- `orders` にユーザー名や住所を入れない  
  → `users` で管理し、`orders.user_id` で紐付け → 3NF を満たす

- `order_items` に商品名や在庫を入れない  
  → `products` で管理

---

### 4. 正規化の結果（オンラインショップ版）

| テーブル    | 内容                                                                |
| ----------- | ------------------------------------------------------------------- |
| users       | ユーザー情報のみ（id, name, email, created_at）                     |
| products    | 商品情報のみ（id, name, price, stock）                              |
| orders      | 注文情報のみ（id, user_id, order_date, status）                     |
| order_items | 注文ごとの商品情報のみ（id, order_id, product_id, quantity, price） |

- 各テーブルは **1NF ～ 3NF** を満たしている
- 多対多関係（注文と商品）は **中間テーブル `order_items`** で整理

---

## 💡 ポイント

- 正規化により **データの重複を減らせる**
- ただし正規化しすぎると **クエリが複雑になる場合がある**
- 実際の運用では、パフォーマンスのために **部分的に冗長化することもある**
